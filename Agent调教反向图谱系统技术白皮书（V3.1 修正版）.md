**3. 核心技术落地细则**

**3.1 AST Diff 算法实现（compareTrees）**

采用 **Tree-sitter 拓扑排序 + Zhang-Shasha 编辑距离**
混合算法，确保结构化差异提取的准确性。

**实现步骤**：

1.  **节点签名与映射**：对新旧 AST 进行后序遍历，为每个节点生成 (type,
    text_hash, children_count)
    三元组签名。优先匹配签名完全相同的子树（O(n) 时间复杂度）。

2.  **差异识别**：对未匹配节点，计算最小编辑距离，输出四类原子操作：

    - UPDATE：节点类型相同但文本变化 → 生成替换规则

    - MOVE：节点签名相同但父节点变化 → 生成结构重排规则

    - INSERT/DELETE：新增或移除节点 → 仅当满足阈值时抽象为规范规则

3.  **规则模板映射**：

4.  **容错降级**：当 Tree-sitter 解析失败或编辑距离 \> 60%
    节点总数时，自动降级为正则匹配，并标记 confidence: low，需人工确认。

**3.2 规则生成阈值与分类**

- **自动生成阈值**：同一模式在 **≥3 个不同文件** 中出现，或 **7 天内重复
  ≥5 次**，才触发规则候选生成。避免一次性修复被误记为规范。

- **优先级体系**：项目级 (1.0) \> 用户级 (0.8) \> 全局级
  (0.5)。高优先级规则自动覆盖低优先级同义规则。

**3.3 MVP 检索策略（确定性匹配）**

**MVP 阶段彻底移除向量语义搜索**，采用纯确定性匹配 +
加权打分，降低工程风险。

**匹配维度**：language + file_extension + rule_tags + project_id +
current_file_path

**相关性打分函数**：

$Score = W_{type} \times TypeWeight + W_{time} \times e^{- \lambda \cdot \Delta t} + W_{match} \times MatchCount$

-  $W_{type} = 0.4,W_{time} = 0.3,W_{match} = 0.3$

-  $\lambda = 0.01$（时间衰减半衰期约 69 小时）

-  $MatchCount$：当前文件路径/名称与规则标签的精确命中数

**Token 控制**：按 Score 降序取 Top-K，严格限制总注入长度 ≤ 2000
tokens。超出部分截断并记录日志，作为 V2.0 开启语义搜索的数据依据。

**4. 交互设计与冲突仲裁**

**4.1 双模式交互**

- **静默模式**：默认开启。后台记录所有修改，仅当满足生成阈值时，在
  Cursor Chat 中以非阻塞消息提示\"已学习新规则\"。

- **确认模式**：专业人士手动开启。每次规则候选均弹出结构化卡片，支持
  \[Yes\] \[No\] \[Edit\] \[Skip\] 四种操作。

**4.2 冲突仲裁界面规范**

当同一作用域内存在 ≥2 条互斥规则时，强制弹出仲裁卡片：

⚠️ **检测到规则冲突**

当前文件 utils.ts 同时命中以下规则：

🔘 保留 A（推荐） 🔘 保留 B 🔘 暂时跳过 🔘 合并为新规则

**决策持久化**：用户选择后，自动生成一条优先级更高的\"仲裁规则\"写入
SQLite，相同冲突不再弹窗。

**批量处理**：支持\"本次会话全部保留 A\"选项，减少打断。
